# 项目级代码维护巡检

## 目标

这个技能用于组织工程师定期回顾自己负责的代码，持续提升项目的稳定性、可维护性与整洁度。

适用目标包括：
- 审查自己近期和历史负责的代码
- 复盘模块边界、职责划分与组织逻辑
- 主动发现 Bug、边界问题、异常流程与潜在回归点
- 删除冗余代码、重复逻辑、失效分支和无用状态
- 简化实现方式，减少不必要的复杂度
- 提炼可复用能力，抽象公共 API 或公共流程
- 统一相似功能的交互、调用链和错误处理方式
- 补充必要验证，保持代码长期整洁、稳定、可扩展

## 何时使用

在以下场景应主动使用本技能：
- 用户要求做一次系统性代码 Review、维护性整理或代码体检
- 某个模块经历多次修补后，逻辑开始分散、重复或难以维护
- 项目进入阶段性收尾、迭代结束前或版本发布前，需要做代码清洁与风险排查
- 出现同类 Bug 反复发生，怀疑存在流程不统一、抽象不足或历史冗余
- 需要推动团队建立定期自查、自测、自维护机制时

## 工作原则

1. 先理解，再修改
- 先梳理模块职责、调用链、状态流转和已有约定
- 不凭表面现象直接重构，优先定位真正的复杂度来源

2. 小步治理，避免误伤
- 以最小必要范围推进整理
- 高风险改动拆分处理，避免一次性大改带来回归

3. 稳定性优先于形式美化
- 优先修复真实缺陷、冗余分支、重复流程和易错结构
- 不为“看起来更优雅”而改动稳定逻辑

4. 抽象必须有收益
- 仅在多个场景存在稳定共性时抽象 API、工具方法或统一流程
- 避免为了抽象而抽象，避免增加理解成本

5. 验证必须闭环
- 每轮清理、重构或修复后，都要做与改动范围匹配的验证
- 验证要覆盖核心路径、风险路径和容易回归的交互

6. 备份优先
- 如果项目已配置 Git，在执行清理代码、删除冗余、批量重构、统一流程前，先检查工作区状态
- 若存在可提交的当前基线，先创建一次备份提交，再开展后续整理
- 未经用户明确要求，不做推送操作

## 执行流程

### 第一步：盘点审查范围

按模块或职责梳理以下内容：
- 当前负责代码的主流程和关键交互
- 重复代码、重复判断、重复状态处理
- 分支过多、命名模糊、职责混乱的位置
- 容易出错的删除、退出、覆盖、异步提交流程
- 缺少统一封装的通用能力
- 失效代码、历史兼容残留、无用函数或无用样式

### 第二步：识别问题类型

重点关注：
- 真实 Bug
- 潜在 Bug
- 冗余实现
- 重复逻辑
- 流程不一致
- API 抽象不足
- 过度耦合
- 可测试性差
- 可读性差但又高频变更

### 第三步：确定治理顺序

建议按以下优先级处理：
1. 会造成错误结果、数据异常、错误交互的问题
2. 高频路径中的重复逻辑和不一致流程
3. 明确无收益的冗余代码和失效分支
4. 能显著提升复用性和可维护性的公共抽象
5. 低风险的结构整理与命名优化

### 第四步：实施整理

实施时遵循以下要求：
- 先消除重复，再考虑抽象
- 先统一行为，再统一写法
- 保持接口语义稳定
- 不随意改变用户可感知行为，除非是在修复缺陷
- 对高风险交互优先统一“确认、执行、反馈”流程
- 删除代码前确认没有实际引用或隐式依赖

### 第五步：完成验证

根据改动范围执行适当验证：
- 运行已有测试、冒烟测试或最小复现脚本
- 手动验证关键主流程和高风险交互
- 检查静态诊断、语法问题、构建问题
- 确认整理后没有引入新的重复逻辑或新的分叉实现

## 输出要求

完成任务后，输出内容应至少包括：
- 本轮审查范围
- 发现的问题分类
- 实际修复或整理的内容
- 删除了哪些冗余
- 抽象了哪些公共能力或统一了哪些流程
- 存在的风险与未处理项
- 已完成的验证方式与结果

## 禁止事项

- 不在未理解原有逻辑前大规模重写
- 不为了追求形式统一而破坏稳定功能
- 不删除用途未确认的代码
- 不跳过验证直接交付
- 不在有 Git 的项目中直接进行大规模清理而不先做备份提交

## 示例

### 示例一：周期性代码体检

用户说：
"帮我对最近这个模块做一次维护性整理，看看有没有冗余逻辑和隐藏 Bug。"

应使用本技能，重点做：
- 盘点模块主流程
- 排查重复逻辑和边界问题
- 清理冗余代码
- 统一关键流程
- 完成验证并汇总结果

### 示例二：多次修补后的流程治理

用户说：
"这个功能修过很多次了，帮我系统整理一下，别再继续堆补丁。"

应使用本技能，重点做：
- 回顾历史补丁堆叠区域
- 识别重复判断和补丁式分支
- 收敛为统一 API 或统一流程
- 补充验证，降低回归风险
